# 要件定義書

## 1. 目的

複数のLLMモデルに対して、同一の開発テーマ・調査テーマを実行し、回答品質・実行時間・コストを比較できるローカルアプリを構築する。

モデル選定を「感覚」や「評判」ではなく、**自社業務に近い評価テーマによる定量比較**で行えるようにすることを目的とする。

### 解決したい課題

- モデル選定の属人化を防ぐ
- SNSや評判ベースの判断を減らす
- 自社業務に近いテーマで比較できる
- モデルごとの得意・不得意を可視化できる
- コストと品質のバランスを比較できる
- 新モデル登場時に再評価できる

---

## 2. 前提条件

### 2.1 インフラ前提

- アプリ本体は個人PC上で動作する
- クラウド上にサーバー・DB・ストレージは構築しない
- データはローカルPC内に保存する
- LLM推論のみ外部APIを利用する

### 2.2 利用する外部API

- OpenAI API
- Anthropic API
- Google Gemini API
- その他 OpenAI互換API

### 2.3 保存データ

以下の情報をローカルDBに保存する。

- 評価テーマ
- 実行対象モデル
- 各モデルの回答
- 実行時間
- トークン数
- 推定コスト
- Judge LLMによる採点結果

---

## 3. 機能要件

### 3.1 評価テーマ管理

ユーザーは評価用テーマを登録・編集・削除できる。

| 登録項目 | 必須 | 説明 |
| -------- | :--: | ---- |
| タイトル | ○ | テーマ名 |
| カテゴリ | ○ | 開発 / 調査 / 設計 / 障害解析 / ドキュメント作成 / コードレビュー / セキュリティ調査 など |
| プロンプト本文 | ○ | 各モデルへ送信する評価プロンプト |
| 評価観点（Rubric） | - | 採点時に重視する観点。JSON保持可 |
| 想定する成果物 | - | 期待されるアウトプット |
| 備考 | - | 補足メモ |

### 3.2 モデル管理

比較対象となるLLMモデルを登録する。

| 登録項目 | 必須 | 説明 |
| -------- | :--: | ---- |
| プロバイダー | ○ | OpenAI / Anthropic / Google / OpenAI互換 |
| モデル名 | ○ | API上のモデル識別子 |
| 表示名 | ○ | 画面表示用の名称 |
| APIエンドポイント | - | OpenAI互換API利用時のベースURL |
| 入力単価 | - | 1Kトークンあたりの入力単価 |
| 出力単価 | - | 1Kトークンあたりの出力単価 |
| 有効 / 無効 | ○ | 実行対象に含めるか |

### 3.3 ベンチマーク実行

ユーザーが評価テーマと対象モデルを選択し、一括実行する。

処理フロー:

1. 評価テーマを取得
2. 対象モデル一覧を取得
3. 各LLM APIへ同一プロンプトを送信（並列・同時実行数制限あり）
4. 回答を保存
5. 実行時間を保存
6. トークン数を保存
7. 推定コストを保存
8. Judge評価へ進む

### 3.4 Judge LLMによる採点

各モデルの回答を、固定したJudge LLMで採点する。評価基準は100点満点。

| 評価項目 | 配点 |
| -------- | -: |
| 正確性 | 40 |
| 網羅性 | 20 |
| 実用性 | 20 |
| 可読性 | 10 |
| 根拠の明確さ | 10 |

Judge LLMはJSON形式で結果を返す。

```json
{
  "accuracy": 36,
  "coverage": 18,
  "practicality": 17,
  "readability": 8,
  "evidence": 7,
  "total": 86,
  "comment": "実装方針は妥当だが、例外処理とコスト面の説明が不足している。"
}
```

### 3.5 結果表示

- **モデル別スコア**: 総合点・実行時間・推定コストの一覧
- **カテゴリ別ランキング**: カテゴリごとの1位モデル
- **詳細比較**: 各モデルの回答全文・Judgeコメント・評価項目別スコア・実行ログ・エラー内容

### 3.6 CSV出力

評価テーマ・モデル名・回答・スコア・実行時間・コスト・JudgeコメントをCSVで出力できる。

## 4. 非機能要件

| 区分 | 要件 |
| ---- | ---- |
| 性能 | 同時実行数は最大3。レート制限・コスト暴発・PC負荷を抑制する |
| 可用性 | ローカル動作のため単一PCで完結。サービス稼働率の概念は対象外 |
| セキュリティ | APIキーは `.env.local` で管理。DB保存・Git管理・画面平文表示・ログ出力を禁止。詳細は[セキュリティ設計書](./07_security-design.md) |
| 保守性 | LLMクライアントは共通インターフェース化し、プロバイダー追加を容易にする |
| 移植性 | データはローカルSQLiteファイル。手動エクスポートでバックアップ可能 |
| 国際化 | UIは日本語を基本とする |

---

## 5. エラー要件

| エラー | 対応 |
| ------ | ---- |
| APIキー未設定 | 設定画面で警告表示 |
| レート制限 | 一定時間待機後リトライ |
| モデル名不正 | エラーとして保存 |
| JSONパース失敗 | `raw_judge_response` を保存 |
| 通信失敗 | 対象モデルのみ失敗扱い |
| 一部モデル失敗 | 他モデルの結果は保存 |

## 6. MVPスコープ

### 6.1 必須機能（MVP）

- 評価テーマ登録
- モデル登録
- 一括実行
- 回答保存
- Judge採点
- 結果一覧表示
- CSV出力

### 6.2 後回し機能（将来拡張）

- ユーザー管理 / 認証
- クラウド同期 / チーム共有
- 自動スケジューリング
- GitHub連携
- ELOレーティング
- 複数回実行による平均点算出
- Judge複数化（GPT Judge + Claude Judge）
- 実案件テンプレート

## 7. 用語定義

| 用語 | 定義 |
| ---- | ---- |
| 評価テーマ（Case） | 各モデルへ送信する評価用プロンプトと評価観点のセット |
| モデル（Model） | 比較対象となるLLM。プロバイダー・モデル名・単価を持つ |
| 実行（Run） | 1つの評価テーマを複数モデルに対して実行する単位 |
| 実行結果（Run Result） | 1モデルの1回答。回答・実行時間・トークン・コストを持つ |
| スコア（Score） | Judge LLMによる採点結果 |
| Judge LLM | 回答を採点する固定のLLM |
| Rubric | 採点の評価基準・観点 |
