# テスト計画書

関連: [要件定義書](./01_requirements.md) / [API設計書](./05_api-design.md) / [実装計画書](./06_implementation-plan.md)

---

## 1. テスト方針

- MVPの機能要件・エラー要件・セキュリティ要件が満たされることを確認する。
- 外部LLM APIはコスト・非決定性があるため、ユニット/結合テストでは**モック**を基本とし、実APIは限定的なスモークテストでのみ使用する。

---

## 2. テストレベル

| レベル | 対象 | 手段 |
| ------ | ---- | ---- |
| ユニット | `cost.ts` / `judge.ts`（JSONパース）/ `csv.ts` / LLM共通IF | モック・純粋関数テスト |
| 結合 | API Routes（cases/models/runs/export） | DBはテスト用SQLite、LLMはモック |
| E2E（手動） | 画面操作フロー | ブラウザ手動 |
| スモーク | 実LLM APIへの最小実行 | 実キーで1テーマ×1モデル |

---

## 3. テスト観点とケース

### 3.1 評価テーマ管理

| # | ケース | 期待 |
| - | ------ | ---- |
| C-01 | 必須項目を入れて登録 | 201・一覧に表示 |
| C-02 | タイトル空で登録 | 400 VALIDATION_ERROR |
| C-03 | 編集して保存 | 200・値が更新 |
| C-04 | 削除 | 200・一覧から消える |
| C-05 | カテゴリで絞り込み | 該当のみ表示 |

### 3.2 モデル管理

| # | ケース | 期待 |
| - | ------ | ---- |
| M-01 | モデル登録 | 201 |
| M-02 | 必須欠落で登録 | 400 |
| M-03 | 有効/無効トグル | 200・状態反映 |
| M-04 | 無効モデルは実行対象選択に出ない | 選択肢に非表示 |

### 3.3 ベンチマーク実行

| # | ケース | 期待 |
| - | ------ | ---- |
| R-01 | 3モデル一括実行（モック成功） | 3件のresult保存・status=done |
| R-02 | 1モデルが失敗 | 失敗モデルは error 保存、他は success |
| R-03 | 全モデル失敗 | run.status=error |
| R-04 | 同時実行数制限 | 同時実行が3以下（p-limit検証） |
| R-05 | タイムアウト | 該当モデルのみ error、他は保持 |
| R-06 | レート制限(429) | バックオフ後リトライ |
| R-07 | latency/tokens/cost が保存される | 値が記録 |

### 3.4 Judge採点

| # | ケース | 期待 |
| - | ------ | ---- |
| J-01 | 正常JSON応答 | 各項目・total・comment保存 |
| J-02 | JSON以外の応答 | raw_judge_response保存・スコアnull |
| J-03 | 余計なテキスト混じりJSON | JSON抽出してパース成功 |
| J-04 | 項目合計とtotalの乖離 | 警告ログ（採用値の方針どおり） |

### 3.5 結果表示・ダッシュボード

| # | ケース | 期待 |
| - | ------ | ---- |
| V-01 | 実行詳細でモデル比較表示 | 表・チャート・回答全文・コメント表示 |
| V-02 | 失敗モデルの表示 | エラー内容を表示 |
| V-03 | ダッシュボードのカテゴリ別ランキング | 各カテゴリ1位を表示 |

### 3.6 CSV出力

| # | ケース | 期待 |
| - | ------ | ---- |
| E-01 | CSVダウンロード | text/csv・列順が仕様どおり |
| E-02 | 回答内の改行・カンマ | 正しくエスケープ（ダブルクォート） |
| E-03 | 失敗モデルの行 | スコア空欄でも出力 |

### 3.7 コスト計算（ユニット）

| # | ケース | 期待 |
| - | ------ | ---- |
| CO-01 | input/output×単価 | `(in/1000)*inCost + (out/1000)*outCost` |
| CO-02 | 単価未設定 | 0 または null として安全に処理 |

### 3.8 セキュリティ

| # | ケース | 期待 |
| - | ------ | ---- |
| S-01 | APIキー未設定で実行 | API_KEY_MISSING・設定画面誘導 |
| S-02 | 設定画面表示 | キー値が平文表示されない（設定済/未設定のみ） |
| S-03 | DB/ログにキーが出ない | grep等でキーが含まれない |
| S-04 | `.gitignore` | `data/`・`.env.local` が除外 |

## 4. 受け入れ基準（MVP）

[実装計画書 §6 完了の定義](./06_implementation-plan.md#6-完了の定義mvp-dod) のチェックリストを全て満たすこと。加えて:

- 主要正常系（C-01, M-01, R-01, J-01, V-01, E-01）が全て成功。
- セキュリティ観点（S-01〜S-04）が全て成功。

## 5. テスト環境

| 項目 | 内容 |
| ---- | ---- |
| OS | 開発者のローカルPC（Windows等） |
| DB | テスト用SQLite（本番DBと分離） |
| LLM | ユニット/結合はモック、スモークのみ実API |
| データ | テスト用シードデータを投入 |

---

## 6. 非テスト範囲（MVP）

- 認証・認可（機能対象外）
- 大規模負荷・並列スケール
- クラウド同期・チーム共有
