# LLM Tactical Arena

ローカル実行型 LLM ベンチマークアプリ。複数のLLMモデルへ同一の評価テーマを実行し、回答品質・実行時間・コストを定量比較します。UIは**未来的なコントロールルーム（管制コンソール）風**のHUDテーマです。

設計ドキュメントは [.docs/](./.docs/README.md) を参照してください。

## 技術スタック

- Next.js 14 (App Router) / React / TypeScript
- Tailwind CSS / Recharts
- Prisma ORM / SQLite（ローカル `./data/benchmark.db`）
- OpenAI / Anthropic / Google Generative AI SDK
- p-limit / Promise.allSettled / AbortController（実行制御）

## セットアップ

```bash
# 1. 依存インストール
npm install

# 2. DB作成（スキーマ反映）。DBパスは .env（コミット済み）から読まれます
npx prisma db push

# 3. 初期データ投入（モデル14件＋評価テーマ15件）
npm run seed

# 4. APIキー設定
cp .env.local.example .env.local
#   .env.local を編集して各APIキーを設定

# 5. 開発サーバー起動
npm run dev
# → http://localhost:3000
```

## 環境変数

| ファイル | 用途 | Git |
| -------- | ---- | --- |
| `.env` | `DATABASE_URL`（DBパス・非機密） | コミット対象 |
| `.env.local` | APIキー（`OPENAI_API_KEY` 等）・`MAX_CONCURRENCY` | **コミット禁止**（`.gitignore` で除外） |

```env
# .env.local
OPENAI_API_KEY=
ANTHROPIC_API_KEY=
GOOGLE_API_KEY=
MAX_CONCURRENCY=3
```

- APIキーはサーバーサイドでのみ参照し、画面・DB・ログには出力しません。
- 設定画面（/settings）では「設定済み/未設定」のみ表示します。

## 使い方

複数のLLMに同じ評価テーマを実行 → 回答をJudge LLMが採点 → スコア・採点根拠・コストを比較する、という流れで使います。

### 1. APIキーを設定する（`/settings`）

`.env.local` に使うプロバイダーのAPIキーを記載し、開発サーバーを再起動します。`/settings` で各プロバイダーが「CONFIGURED / MISSING」のどちらかを確認できます（キーの値は表示されません）。未設定のプロバイダーのモデルは実行時にブロックされます。

### 2. 対象モデルを確認・登録する（`/models`）

シードで主要モデル14件が登録済みです。各モデルの **プロバイダー / モデル名（API識別子）/ 入力・出力単価 / 有効・無効** を確認し、必要に応じて編集・追加します。

- `ENABLED / DISABLED` トグルで実行対象に含めるかを切り替え（無効モデルは実行時の選択肢に出ません）。
- 単価はコスト推定に使われます。**OpenAI / Google の初期単価は概算なので、実際の料金に更新してください**（Claudeは公式値）。
- OpenAI互換APIを使う場合は、プロバイダーに「OpenAI互換」を選び、APIエンドポイント（ベースURL）を指定します。

### 3. 評価テーマを登録する（`/cases`）

シードで15テーマが登録済みです。`+ NEW CASE` から自社業務に近いテーマを追加できます。

- **タイトル / カテゴリ / プロンプト本文**（必須）、評価観点・想定成果物・備考（任意）。
- プロンプト本文が各モデルへそのまま送信されます。

> ⚠️ プロンプトは外部LLM APIへ送信されます。顧客名・個人情報・認証情報などはマスキングしてください。

### 4. ベンチマークを実行する（テーマ詳細 → `RUN BENCHMARK`）

テーマ詳細画面で以下を選び、`▶ RUN BENCHMARK` を押します。

- **対象モデル**: 比較したいモデルを複数選択。
- **Judgeモデル**: 回答を採点するモデルを1つ選択。

各モデルへ並列でプロンプトを送信（同時実行数は既定3）し、回答・実行時間・トークン数・推定コストを保存。続けてJudge LLMが各回答を100点満点で採点します。完了すると実行詳細へ遷移します。

### 5. 結果を読む（`/runs/[id]`）

実行詳細画面で以下を確認できます。

- **EVALUATION TASK**: 送信したプロンプト全文・想定成果物。
- **SCORING RUBRIC**: 採点の配点（正確性40 / 網羅性20 / 実用性20 / 可読性10 / 根拠10＝100点）と各観点の意味。
- **SCORE MATRIX / CHART**: モデル別の項目別スコア・総合点・時間・コストの一覧と棒グラフ（1位をハイライト）。
- **JUDGE RATIONALE（採点根拠）**: モデルごとに、項目別スコアのバー＋**Judgeコメント（採点理由）**を表示。`▶ 回答を確認する` で回答全文と実行メトリクスを開閉。
- 一部モデルが失敗しても、他モデルの結果は保存されます（失敗モデルはエラー内容を表示）。
- `⬇ CSV EXPORT` でテーマ・回答・スコア・時間・コスト・JudgeコメントをCSV出力。

### 6. 全体を俯瞰する（`/dashboard`）

複数回実行した結果を集計し、**モデル別の平均スコア・平均時間・累計コスト**と**カテゴリ別の1位モデル**を表示します。実行履歴は `/runs` でテーマ詳細付きのカード一覧として確認できます。

### 採点基準（Judge LLM）

| 評価項目 | 配点 | 観点 |
| -------- | -: | ---- |
| 正確性 | 40 | 事実・技術的な正しさ |
| 網羅性 | 20 | 要求事項のカバー範囲 |
| 実用性 | 20 | 実務での使いやすさ・具体性 |
| 可読性 | 10 | 構成・説明の分かりやすさ |
| 根拠の明確さ | 10 | 主張の裏付け・出典 |

## 注意

- 評価プロンプトは外部LLM APIへ送信されます。機密情報はマスキングしてください。
- データはローカルSQLiteに保存されます。バックアップはDBファイル（`data/benchmark.db`）の手動コピーで行ってください。
- `npm run seed` は既存データを全削除して初期データを再投入する**クリーンリセット**です。自分で登録したテーマ・モデルがある場合は実行に注意してください。
